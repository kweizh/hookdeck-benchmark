Upstream producers send Stripe-shaped webhooks where the meaningful payload is nested under `body.data.object`. The downstream service expects a flattened payload, a renamed `email` field, a hint header, and only events whose `amount` is at or above a configured threshold. You must implement this end-to-end on Hookdeck using a JavaScript Transformation rule. ## Requirements * Create a Hookdeck **Source** of type `WEBHOOK`. * Create a Hookdeck **Destination** of type `MOCK_API` (so events are accepted and recorded by Hookdeck without needing a public inbound URL). * Create a Hookdeck **Transformation** (its JavaScript code lives in Hookdeck and is referenced from the Connection by its `transformation_id`). The transformation contract is: * **Input shape** (what the source receives): ```json { "data": { "object": { "id": "<string>", "amount": <number>, "currency": "<string>", "customer_email": "<string>" } } } ``` * **Output body shape** (after transformation, when delivered): ```json { "id": "<string>", "amount": <number>, "currency": "<string>", "email": "<string>" } ``` * **Output headers** must include `x-hookdeck-transformed: true` (in addition to whatever headers Hookdeck normally appends). * **Drop rule**: if the input `body.data.object.amount` is strictly less than `100`, the event MUST NOT be delivered. (Returning `null` from the handler tells Hookdeck to drop the event.) * Create a Hookdeck **Connection** that links the Source to the Destination and applies the Transformation as a rule referenced by its `transformation_id`. * Publish exactly **three** test events to the Source. They must have these `data.object.amount` values, in this order: `50`, `200`, `200`. Use distinct `id` and `customer_email` values so they can be distinguished in the inspector. * Record the IDs and a small JSON summary in a log file so the verifier can find them. ## Implementation Hints * Authenticate the CLI in headless mode with `hookdeck ci --api-key $HOOKDECK_API_KEY` before any CLI calls, OR drive everything through the REST API at `https://api.hookdeck.com/2025-07-01` using `Authorization: Bearer $HOOKDECK_API_KEY`. * The Transformations sandbox is a V8 isolate. It does **not** support `fetch`, `async/await`, file I/O, or any external network access. Only `JSON.parse` and `JSON.stringify` are guaranteed beyond standard ES6 language features. * The transformation handler must be registered with `addHandler("transform", (request, context) => { ... })`. Mutating `request.body` and `request.headers` and returning the modified `request` is the supported pattern. Returning `null` drops the event. * The Transformation is created via `POST /transformations` and then referenced from a Connection via a rule of `{ "type": "transformation", "transformation_id": "trs_..." }`. * For a `MOCK_API` destination, set `destination.type` to `MOCK_API`; Hookdeck will accept the delivery and you can read it back through the Inspect API (`GET /events`, `GET /events/:id`). * To publish test events, you may either POST directly to the Source's ingest `url` (returned when the source is created) or use the Publish API at `https://hkdk.events/v1/publish` with the `X-Hookdeck-Source-Name` header.
